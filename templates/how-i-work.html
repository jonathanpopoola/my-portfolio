How I work

At the start of any project I always start with two questions, the answers from these two questions form the basis of all decisions taken during a project. These two questions are simple enough, "what are the business/ stake holders trying to achieve ?" and "what is the end user trying to achieve ?".

As simple as these questions maybe, the answers to these questions tend to be alot more complex especially when we consider the end user. In a simple world we could mearly answer these questions with "business/ stake holders want to sell products/ more products" and the user "wants to buy products". However these questions offer us no insight to the real needs of the end user, we still have many questions to answer for example what prompts the user to consider a purchase, what are their concerns during this purchase ? These two questions also poise a new question, how do we couple business needs with those of the user ?

User centered design

To effectively answer these questions and architect a great user experience I follow a user centered design process. The exact process I take depends on real world constraints of time, budget and availbilty of resources. In all cases the process taken is an iterative process involving a number of stages as illustrated below:

As you can see the process I follow consists of three key stages which I will explain in detail below.

Tools I use to craft

Research

The first phase in my iterative UCD process is the research stage, it is during this phase when I really try to get to know the users. Research is not limited to just the intial stage there are times when it is appropriate to revisit certain research tasks/ documents. For example User Personas should be a living document and kept up to date, with any information discovered along the process.

TaskPurposeDeliverables
User workshopsUser workshops are usually one of the first research tasks I perform, they are usually informal sessions with a group of users. They tend to be informal as this creates a more relaxed setting, with relaxed users they tend to open up and offer more information. The main purpose of these workshops differ slightly depending on a number of factors. However the rough goals are always the same. To find out what the user wants to achieve, how they currently go about this task, current pains with the task and any areas of improvements they feel would help.

Design

Once sufficient research has taken place, the design of the user experience then begins. The research not only provides a base for the design, but is used also to validate the user experience. For example user journeys should be validated against task models to ensure that all requirements are met.

As mentioned I follow an iterative UCD process

Once wireframes have been generated I will then create low fedilty prototypes, there are a number of reasons for this. The first being that usabilty tests can be performed, as well as conveying ideas more effectively. Low fedility prototypes can also be used to gain sign off from stake holders as well as vslidating designs and ideas. In some cases low fedilty wireframes are not enough to convey ideas so high fedilty prototypes can be created. I have created a framework for high fiedlity prototyping using html5, Bootstrap and JavaScript that allows me to rapidly create high fidelity prototypes. High fidelity prototypes are also a great way of demonstrating UI requirements to developers ensuing the system meets UX requirements exactly.

Evaluate

As mentioned the user centred design process I follow is an iterative cycle therefore evaluation occurs frequently. All deliverables from the design stage will be evaluated against the requirements gathered in the research phase.

The high fedilty prototypes I create are used to visualise the end result of the proposed solution, they are also critical in ensuring that user requirements are met. I like to perform a number of usabilty tasks with the created high fedilty prototypes.

I perform a number of user observation tasks, setting a user a number of goals to perform using the prototype. Then observing how the user executes the task within the prototype. This is essential in ensuring the users needs have been met and revealing other important information such as points they struggled with, points that worked well etc.

I also like to perform A and B testing, providing a number of variations within the prototype to analyse which performs more effectively.
